Ce que j'ai découvert sur CyberStrikeAI

1) C'est un outil offensif activement utilisé dans des cyberattaques réelles. Guide d'intégration étape par étape
Étape 1 — Extraire le zip
Extraire queen_v1_patterns.zip dans ton dossier queen_v1/ existant. Les fichiers se placent automatiquement aux bons endroits.

Étape 2 — Ajouter PyYAML aux dépendances
Dans queen_core/requirements.txt, ajouter :

pyyaml>=6.0,<7.0
Et dans dashboard/backend/requirements.txt, même chose.

Étape 3 — Monter tools/, roles/, skills/ dans Docker
Dans docker-compose.yml, ajouter les volumes à orchestrator, worker-1, et dashboard :

yaml
    volumes:
      # ... volumes existants ...
      - ./tools:/app/tools:ro
      - ./roles:/app/roles:ro
      - ./skills:/app/skills:ro
Le :ro signifie read-only — les containers ne peuvent pas modifier les définitions.

Étape 4 — Brancher le tool_router dans l'API
Dans dashboard/backend/api.py, après la ligne app.add_middleware(...), ajouter :

python
from dashboard.backend.tool_routes import tool_router
app.include_router(tool_router)
Ça expose automatiquement les endpoints GET /api/tools, GET /api/tools/{name}, POST /api/tools/reload, etc.

Étape 5 — Utiliser les rôles dans le planner
Dans queen_core/planner.py, modifier la fonction de planning pour charger le rôle si le goal en a un. Ajouter au début du fichier :

python
from queen_core.role_registry import get_role_registry
Puis dans la logique de planning, avant l'appel au LLM :

python

if role_name:

        # Utiliser ce system_prompt dans l'appel LLM
        # + filtrer les outils autorisés via role.tools
Étape 6 — Utiliser les tools dans le worker
Dans workers/worker_unified.py, au lieu d'avoir les timeouts hardcodés, charger depuis le registre. Ajouter :

python
from queen_core.tool_registry import get_registry
Puis dans le handler de job :

python
tool_reg = get_registry()
tool_def = tool_reg.get("python_exec")  # ou selon le job_type
if tool_def:
    timeout = tool_def.timeout_seconds
    blocked = tool_def.blocked_patterns
Étape 7 — Donner les skills au LLM
Dans le worker, quand on construit le prompt LLM pour un job de type codegen ou research, charger les skills pertinents :

python
from queen_core.skill_registry import get_skill_registry

skill_reg = get_skill_registry()

# Le rôle définit les skills recommandés
for skill_name in role.skills:  # ex: ["python-optimization", "testing-strategies"]
    content = skill_reg.read_skill(skill_name)
    if content:
        # Ajouter au contexte LLM (dans le system prompt ou en user message)
        prompt += f"\n\n--- Skill: {skill_name} ---\n{content}"
Étape 8 — Ajouter un rôle au goal (frontend)

Étape 10 — Ajouter de nouveaux outils/rôles/skills
Pour ajouter un outil : créer un fichier tools/mon_outil.yaml et faire POST /api/tools/reload.

Pour ajouter un rôle : créer roles/mon_role.yaml et redémarrer.

Pour ajouter un skill : créer skills/mon-skill/SKILL.md et redémarrer.

Aucun code Python à modifier pour étendre le système.
