J'ai exploré le catalogue Aixploria et identifié les projets GitHub pertinents pour Queen V1. La plupart des 143 projets listés sont des outils de génération vidéo, image ou voix — pas notre domaine. Mais 5 projets ont des patterns architecturaux directement transposables.

Projets pertinents pour Queen V1
1. AutoGPT — Boucle autonome Goal → Plan → Execute → Evaluate
C'est l'ancêtre de ce que Queen V1 fait déjà. Mais AutoGPT a mûri depuis 2023 et a ajouté deux choses qu'on n'a pas :

Mémoire long-terme (vector store). Actuellement Queen V1 oublie tout entre les runs. Si un goal "optimise le scoring" a déjà été tenté et a échoué, le LLM ne le sait pas. AutoGPT stocke les résultats passés dans une base vectorielle et les injecte comme contexte.

Ce qu'on intégrerait : un module queen_core/memory_vector.py qui utilise ChromaDB (léger, sans serveur) pour indexer les résultats de chaque run. Avant de planifier un nouveau goal, le planner recherche les runs similaires passés et injecte les leçons apprises dans le prompt. Ça évite que le système répète les mêmes erreurs.

2. Emergent AI — Architecture Multi-Agents spécialisés
Leur pattern clé : au lieu d'un seul agent qui fait tout, ils ont un agent Frontend, un agent Backend, un agent Tests, qui travaillent en parallèle et se coordonnent. Chaque agent a son propre contexte.

Ce qu'on intégrerait : nos workers sont déjà séparés par type (research, codegen, test, eval, patch) mais ils utilisent tous le même prompt. On ajouterait un agent prompt par job type — chaque worker construit un prompt spécialisé en combinant le rôle + le skill + le contexte des jobs précédents du même run. Concrètement, le worker codegen recevrait automatiquement le résultat du worker research dans son prompt.

3. SuperAGI — Gestion de ressources et monitoring d'agents
SuperAGI a un bon système de monitoring : tokens consommés par run, coût estimé, temps d'exécution, et surtout un budget par agent qui stoppe l'exécution si le LLM boucle.

Ce qu'on intégrerait : un module queen_core/budget_tracker.py qui compte les tokens et le temps par run. Si un run dépasse son budget (configurable dans le rôle YAML), il s'arrête proprement au lieu de tourner en boucle. On afficherait aussi les coûts dans le dashboard (page Settings ou Overview).

4. OpenManus — Chaînage de résultats entre étapes
OpenManus passe automatiquement l'output d'une étape comme input de la suivante, avec un résumé si l'output est trop long. C'est simple mais on ne le fait pas encore — nos jobs sont indépendants, chaque worker repart de zéro.

Ce qu'on intégrerait : modifier l'orchestrator pour que quand un job step 2 dépend de step 1, le résultat de step 1 soit automatiquement injecté dans le payload de step 2. Avec un summarize_if_too_long() si le résultat dépasse 2000 tokens.

5. BabyAGI — Replanification dynamique
BabyAGI ne se contente pas d'exécuter un plan fixe. Après chaque étape, il réévalue si le plan est encore pertinent et peut ajouter/supprimer des étapes. Notre planner génère un plan au début et ne le modifie plus.

Ce qu'on intégrerait : un replan check après chaque job terminé. L'orchestrator demande au LLM "le plan est-il encore valide vu ce résultat ?" et peut ajouter un job ou en supprimer un. Ça rend le système adaptatif au lieu de rigide.

2	Budget tracker (tokens + temps + coût)	SuperAGI	Empêche les boucles infinies, donne de la visibilité sur les coûts	Moyen — nouveau module + dashboard

Amélioration #1 — Chaînage des résultats (OpenManus)
Fichier : queen_core/job_chain.py

Quand l'orchestrator dispatch un job step 3, il récupère automatiquement les résultats des steps 1 et 2 et les injecte dans le payload sous la clé previous_context. Si le contexte dépasse 4000 caractères, il est compressé automatiquement (garde seulement les summaries et les noms de fichiers).

Ce qui change dans l'orchestrator : une seule ligne ajoutée dans _dispatch_jobs — job = enrich_job_payload(memory, job) — juste avant redis_bus.enqueue_job(job).

Amélioration #4 — Prompts spécialisés (Emergent AI)
Fichier : queen_core/prompt_builder.py

Chaque type de job a maintenant son propre system prompt et son propre user template. Le worker research reçoit un prompt d'analyste, le codegen un prompt de développeur senior, le eval un prompt d'évaluateur exigeant. Les prompts sont enrichis avec le rôle YAML (si défini), les skills chargés à la demande, et le contexte cumulé des étapes précédentes.

Ce qui change dans le worker : les handlers handle_research, handle_codegen et handle_eval n'ont plus de prompts hardcodés. Ils appellent build_prompt(job, role_name, extra_skills) qui construit tout dynamiquement.

Amélioration #2 — Budget Tracker (SuperAGI)
Fichier : queen_core/budget_tracker.py

Chaque appel LLM est automatiquement comptabilisé : tokens input/output, nombre d'appels, coût estimé, temps écoulé. On peut vérifier à tout moment si un run a dépassé son budget (tokens, temps, coût, nombre d'appels).

Ce qui change dans llm_client.py : la fonction generate() accepte un paramètre optionnel run_id. Si fourni, elle enregistre les tokens dans le tracker. Les backends Ollama et OpenAI retournent maintenant les vrais token counts de l'API (Ollama : prompt_eval_count/eval_count, OpenAI : usage.prompt_tokens/usage.completion_tokens).
