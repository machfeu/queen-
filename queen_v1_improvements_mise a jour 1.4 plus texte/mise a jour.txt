Projets gratuits pertinents et ce qu'on intègre
Sur les 5000+ outils listés, j'ai filtré ceux qui ont un pattern architectural transposable à Queen V1. La plupart (vidéo, musique, image, SEO) ne nous concernent pas. Voici les 6 projets utiles et pour chacun, exactement ce qu'on prend :

1. OpenHands (65K+ stars, MIT) — Event Stream Architecture
C'est le plus important. OpenHands a résolu un problème qu'on a : la communication agent↔sandbox est fragile quand elle passe par des queues Redis classiques. Leur solution : un Event Stream — tout (actions, observations, résultats, erreurs) est un événement immutable dans un flux ordonné. Ça donne un historique complet, rejouable, auditable.

Ce qu'on prend : Un module event_stream.py qui remplace notre communication Redis fire-and-forget par un log d'événements persistent. Chaque action du worker, chaque résultat, chaque erreur est un événement horodaté et stocké. Ça permet de rejouer un run, de debugger, et de reprendre après un crash.

2. Grok 4.20 (multi-agents correcteurs) — Consensus entre agents
La description Aixploria dit : "quatre agents de niveau professeur qui raisonnent ensemble avant de répondre, se corrigent mutuellement". C'est exactement ce qui manque à notre phase eval : un seul LLM évalue son propre travail.

Ce qu'on prend : Un module consensus.py — au lieu qu'un seul appel LLM évalue un résultat, on fait N appels indépendants (avec des prompts légèrement différents) et on agrège les scores. Si 2 évaluateurs sur 3 rejettent, le patch est rejeté. Ça élimine les cas où le LLM valide son propre code médiocre.

3. OpenClaw (agent proactif, MIT) — Notifications et heartbeat
OpenClaw a un concept intéressant : l'agent prend l'initiative de communiquer (via Signal/Telegram/etc.) au lieu d'attendre qu'on le sollicite. Pour Queen V1, ça se traduit par des notifications proactives : le système envoie un message quand un patch est prêt à approuver, quand un run échoue, quand le budget est dépassé.

Ce qu'on prend : Un module notifier.py avec un système de webhooks configurable. Quand un événement important se produit (patch prêt, run terminé, budget dépassé), Queen envoie un POST vers un webhook (Slack, Discord, Telegram, email).

4. AutoGen (Microsoft, open source) — Conversation multi-agents structurée
AutoGen organise les agents en "groupes de conversation" où chaque agent a un rôle et peut déléguer à un autre. Notre orchestrator fait tout séquentiellement. AutoGen montre qu'un agent research devrait pouvoir demander un complément d'info à un agent codegen en cours de route.

Ce qu'on prend : Un système de feedback loop entre steps — si le worker test trouve des erreurs, au lieu de juste les logger, il peut créer un job codegen_fix automatiquement pour corriger. C'est le self-healing d'Emergent AI combiné avec la délégation d'AutoGen.

5. Emergent AI — Self-Healing (auto-correction)
Décrit dans Aixploria : "Si une erreur survient lors de la compilation, l'agent la détecte, analyse le bug, propose un correctif et l'applique sans intervention". Notre pipeline actuel s'arrête si un test échoue.

Ce qu'on prend : Intégré dans le feedback loop ci-dessus — un module self_heal.py qui intercepte les échecs de test/syntax, génère un job de correction automatique, et relance le test. Maximum 2 tentatives avant escalade humaine.

6. SERA / AI2 Open Coding Agents — Évaluation SWE-Bench
Leur innovation : ils génèrent des paires correct/incorrect pour entraîner l'évaluation. On ne peut pas entraîner un modèle, mais on peut utiliser leur pattern pour améliorer nos prompts d'évaluation : donner des exemples de bon et mauvais code dans le prompt eval.

Ce qu'on prend : Des few-shot examples dans nos skills d'évaluation — le skill code-evaluation contiendra des exemples de code bon et mauvais avec les scores attendus, pour calibrer le LLM évaluateur.

#2 — Consensus Multi-Eval (Grok 4.20)
queen_core/consensus.py — 3 évaluateurs indépendants avec des perspectives différentes :

correctness_expert : le code fait-il ce qui est demandé ?
security_reviewer : y a-t-il des failles ?
maintainability_judge : le code est-il maintenable ?
Le score final est la médiane (résistant aux outliers). Si 2 évaluateurs sur 3 approuvent → approve. Si les scores divergent de plus de 0.3 → notification de désaccord.
